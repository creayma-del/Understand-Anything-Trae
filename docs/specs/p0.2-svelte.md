# P0.2: Svelte 详细设计规格

> 版本: 4.1
> 日期: 2026-06-04
> 状态: 已评审（修复 Critical 问题后）
> 变更: v4.0 方案升级 — 从"无 Tree-sitter 模式"升级为"独立 Plugin + 官方编译器"方案，使用 svelte/compiler 解析 Svelte 结构 + TypeScript tree-sitter 解析 script 内容，实现确定性结构化分析

## 1. 目标

为项目新增 Svelte (`.svelte`) 的完整支持，包括：
- 语言配置注册（core 包 + scan-project.mjs 双端）
- **确定性结构化提取**（svelte/compiler + TypeScript tree-sitter）
- LLM 分析提示词
- 导入解析增强
- 指纹系统升级（从降级模式升级为结构化分析）

## 2. Svelte 文件结构分析

Svelte 组件是单文件组件，结构与 Vue SFC 类似但语法不同：

```svelte
<script context="module">
  export function helper() { /* ... */ }
</script>

<script>
  import { onMount } from 'svelte'
  import Child from './Child.svelte'

  let count = 0
  $: doubled = count * 2
  $: if (count > 10) { console.log('high') }

  onMount(() => { /* ... */ })
</script>

<div>
  <p>{count} x 2 = {doubled}</p>
  <Child {count} on:click={() => count++} />
</div>

<style>
  p { color: red; }
</style>
```

### 2.1 Svelte 5 Runes 语法

Svelte 5 引入了 Runes，一种新的响应式模型：

```svelte
<script>
  let count = $state(0);
  let doubled = $derived(count * 2);
  let { name, age } = $props();

  $effect(() => {
    console.log(count);
  });
</script>
```

Runes 中的 `$state()`, `$derived()`, `$effect()`, `$props()` 是编译器识别的特殊函数调用，在 tree-sitter 中被解析为普通的函数调用表达式。

**P0.2 处理策略**: Runes 语法作为普通函数调用提取（`$state` → function, `$derived` → function），不需要特殊 AST 处理。LLM 指南中需要覆盖 Runes 语义。

### 2.2 提取策略：官方编译器 + TypeScript tree-sitter

**方案升级说明**: 之前因 tree-sitter-svelte WASM 不可用而降级为纯 LLM 分析。现升级为使用 Svelte 官方编译器 `svelte/compiler` 解析 Svelte 结构，再用 TypeScript tree-sitter 解析 script 内容，实现**确定性**结构化分析。

```
阶段 1: svelte/compiler.parse() 解析 .svelte 文件
  → 获取 AST：script / moduleScript / template / css
  → 提取 script 内容（start 行号用于行号偏移）
  → 提取 template 中的组件引用（大写开头的标签）

阶段 2: TypeScript tree-sitter 解析 script 内容
  → 复用 TreeSitterPlugin 中已加载的 TypeScript grammar
  → 复用 TypeScriptExtractor 提取函数/类/导入/导出/调用图
  → 行号偏移修正：script 节点在 .svelte 文件中的起始行
```

### 2.3 WASM 兼容性验证结论（保留参考）

tree-sitter-svelte 的 WASM grammar 不可用（external scanner 限制），因此：
- LanguageConfig 中**不设 `treeSitter` 字段**（避免 TreeSitterPlugin 尝试加载 WASM）
- 不创建 LanguageExtractor（不依赖 tree-sitter-svelte AST）
- **改为创建独立的 SveltePlugin**，使用 svelte/compiler 替代 tree-sitter-svelte

### 2.4 AST 结构验证结果（已确认）

**Svelte `<script>` 内容为 `raw_text`，与 Vue SFC 相同。**

使用 `tree-sitter parse` 验证的实际 AST 输出：
```
(script_element [0, 0] - [11, 9]
  (start_tag [0, 0] - [0, 8]
    (tag_name [0, 1] - [0, 7]))
  (raw_text [1, 2] - [11, 0])    ← raw_text，非递归解析
  (end_tag [11, 0] - [11, 9]
    (tag_name [11, 2] - [11, 8])))
```

由于使用 svelte/compiler 替代 tree-sitter-svelte，此验证结论仅作参考。

## 4. Layer 2: 结构化提取 — SveltePlugin

### 4.1 架构决策：独立 AnalyzerPlugin

**不使用 TreeSitterPlugin 处理 .svelte 文件**，而是创建独立的 `SveltePlugin`，原因：

1. TreeSitterPlugin 要求 LanguageConfig 包含 `treeSitter` 字段才能加载 grammar
2. tree-sitter-svelte WASM 不可用（external scanner 限制）
3. Svelte 需要先拆分 SFC 结构，再解析 script 内容，这是 TreeSitterPlugin 不支持的两阶段流程
4. 独立插件遵循项目已有的插件模式（MarkdownParser、VueSfcPlugin 等都是独立插件）

### 4.2 SveltePlugin 设计

**文件**: `packages/core/src/plugins/parsers/svelte-parser.ts`

### 4.3 svelte/compiler 解析流程

```typescript
import { parse as parseSvelte } from "svelte/compiler";

function parseSvelteSfc(content: string, filePath: string) {
  const ast = parseSvelte(content, {
    filename: filePath,
    // modern: true, // Svelte 5 模式（如果可用）
  });

  // ast 结构:
  // - module: { content: string, start: number, end: number } | null  (context="module" 的 <script>)
  // - instance: { content: string, start: number, end: number } | null  (实例级 <script>)
  // - html: { children: [...] }  (template AST)
  // - css: { children: [...] } | null  (<style> 块)

  // 提取 script 内容 — 需要同时处理 instance 和 module 两个 script 块
  // instance: 组件实例级代码（props, reactive state, lifecycle）
  // module:  模块级导出代码（可被外部 import 的函数/常量）
  const scriptBlocks: Array<{
    content: string;
    startLine: number;
    isModule: boolean;
  }> = [];

  if (ast.instance) {
    scriptBlocks.push({
      content: ast.instance.content,
      startLine: content.slice(0, ast.instance.start).split("\n").length,
      isModule: false,
    });
  }
  if (ast.module) {
    scriptBlocks.push({
      content: ast.module.content,
      startLine: content.slice(0, ast.module.start).split("\n").length,
      isModule: true,
    });
  }

  // 提取 template 中的组件引用
  const templateComponents = extractTemplateComponents(ast.html, content);

  return { scriptBlocks, templateComponents, ast };
}
```

**关键差异（vs @vue/compiler-sfc）**:
- svelte/compiler 的 `start`/`end` 是字符偏移量（character offset），不是行号
- 需要通过 `content.slice(0, start).split("\n").length` 转换为行号
- `instance` 对应实例级 `<script>`，`module` 对应 `<script context="module">`
- `content` 字段是纯 JS/TS 代码文本，不含 `<script>` 标签

### 4.4 Script 内容的 TypeScript tree-sitter 解析

```typescript
analyzeFile(filePath: string, content: string): StructuralAnalysis {
  const { scriptBlocks, templateComponents } =
    parseSvelteSfc(content, filePath);

  // 如果没有 script 块，仅返回 template 组件引用
  if (scriptBlocks.length === 0) {
    return {
      functions: [],
      classes: [],
      imports: templateComponents.map(comp => ({
        source: comp.name,
        specifiers: [comp.name],
        lineNumber: comp.lineNumber,
      })),
      exports: [],
    };
  }

  // [降级守卫] tsPlugin 未初始化时返回空分析
  if (!this.tsPlugin) {
    return {
      functions: [],
      classes: [],
      imports: templateComponents.map(comp => ({
        source: comp.name,
        specifiers: [comp.name],
        lineNumber: comp.lineNumber,
      })),
      exports: [],
    };
  }

  // 合并所有 script 块的分析结果
  // Svelte 可以同时有 instance script 和 module script
  const isTs = detectTypeScriptScript(content);
  const allFunctions: StructuralAnalysis["functions"] = [];
  const allClasses: StructuralAnalysis["classes"] = [];
  const allImports: StructuralAnalysis["imports"] = [];
  const allExports: StructuralAnalysis["exports"] = [];

  for (const block of scriptBlocks) {
    const tsAnalysis = this.tsPlugin.analyzeFile(
      isTs ? "virtual.ts" : "virtual.js",
      block.content,
    );

    // 行号偏移修正
    const lineOffset = block.startLine - 1;
    allFunctions.push(...tsAnalysis.functions.map(fn => ({
      ...fn,
      lineRange: [fn.lineRange[0] + lineOffset, fn.lineRange[1] + lineOffset] as [number, number],
    })));
    allClasses.push(...tsAnalysis.classes.map(cls => ({
      ...cls,
      lineRange: [cls.lineRange[0] + lineOffset, cls.lineRange[1] + lineOffset] as [number, number],
    })));
    allImports.push(...tsAnalysis.imports.map(imp => ({
      ...imp,
      lineNumber: imp.lineNumber + lineOffset,
    })));
    allExports.push(...tsAnalysis.exports.map(exp => ({
      ...exp,
      lineNumber: exp.lineNumber + lineOffset,
    })));
  }

  // 合并 template 组件引用到 imports
  // 去重：检查组件名是否已出现在 import specifiers 中
  const importedSpecifiers = new Set(allImports.flatMap(i => i.specifiers));
  const templateImports = templateComponents
    .filter(comp => !importedSpecifiers.has(comp.name))
    .map(comp => ({
      source: comp.name,
      specifiers: [comp.name],
      lineNumber: comp.lineNumber,
    }));

  return {
    functions: allFunctions,
    classes: allClasses,
    imports: [...allImports, ...templateImports],
    exports: allExports,
  };
}
```

### 4.5 检测 TypeScript Script

```typescript
/**
 * 检测 .svelte 文件的 <script> 是否使用 TypeScript。
 * svelte/compiler 不直接提供 lang 属性，需要从原始内容中检测。
 */
function detectTypeScriptScript(content: string): boolean {
  const match = content.match(/<script[^>]*\blang\s*=\s*["']ts["']/);
  return match !== null;
}
```

### 4.6 Template 组件引用提取

```typescript
interface TemplateComponentRef {
  name: string;       // 大写开头的组件名，如 "Child"
  lineNumber: number; // 在 .svelte 文件中的行号
}

/**
 * 从 Svelte template AST 中提取组件引用。
 * Svelte 约定大写开头的标签为组件，小写标签为 HTML 元素。
 * 也处理 <svelte:xxx> 特殊元素。
 */
function extractTemplateComponents(
  html: { children: any[] } | null,
  content: string,  // 原始文件内容，用于字符偏移转行号
): TemplateComponentRef[] {
  if (!html) return [];

  const components: TemplateComponentRef[] = [];
  const isComponentTag = (name: string) => /^[A-Z]/.test(name);

  function walk(node: any) {
    if (!node) return;

    // Svelte AST 节点类型
    if (node.type === "Component" || (node.type === "InlineComponent" && isComponentTag(node.name))) {
      components.push({
        name: node.name,
        lineNumber: node.start ? content.slice(0, node.start).split("\n").length : 0,
      });
    }

    // 递归子节点
    if (node.children) {
      for (const child of node.children) {
        walk(child);
      }
    }

    // 处理 if/each/await 块的子节点
    if (node.else) walk(node.else);
    if (node.then) walk(node.then);
    if (node.catch) walk(node.catch);
  }

  if (html.children) {
    for (const child of html.children) {
      walk(child);
    }
  }
  return components;
}
```

**注意**: svelte/compiler 的 AST 中，组件节点的 `start` 也是字符偏移量，需要转换为行号。由于 `extractTemplateComponents` 需要原始内容来计算行号，实际实现中需要传入 `content` 参数。

### 4.7 Call Graph 提取

```typescript
extractCallGraph(filePath: string, content: string): CallGraphEntry[] {
  const { scriptBlocks } = parseSvelteSfc(content, filePath);
  if (scriptBlocks.length === 0 || !this.tsPlugin) return [];

  const isTs = detectTypeScriptScript(content);
  const allCallGraph: CallGraphEntry[] = [];

  for (const block of scriptBlocks) {
    const callGraph = this.tsPlugin.extractCallGraph(
      isTs ? "virtual.ts" : "virtual.js",
      block.content,
    );

    // 行号偏移修正
    const lineOffset = block.startLine - 1;
    allCallGraph.push(...callGraph.map(entry => ({
      ...entry,
      lineNumber: entry.lineNumber + lineOffset,
    })));
  }

  return allCallGraph;
}
```

## 5. Layer 3: LLM 指南

### 5.1 文件: `skills/understand/languages/svelte.md`

```markdown
# Svelte Language Prompt Snippet

## Key Concepts

- **Single-File Components**: `.svelte` files combine script, template, and style
- **Reactive Declarations**: `$: doubled = count * 2` — auto-updates when dependencies change
- **Reactive Statements**: `$: if (count > 10) { ... }` — runs when dependencies change
- **Store Auto-Subscription**: `$storeName` — auto-subscribes and unsubscribes to Svelte stores
- **Component Props**: `export let propName` — declares component inputs
- **Component Events**: `createEventDispatcher()` or Svelte 5 `$emit` — component outputs
- **Context API**: `setContext(key, value)` / `getContext(key)` — dependency injection without props
- **Slots**: Default and named slots for content distribution
- **Actions**: `use:action` — DOM element behavior attachment
- **Transitions**: `transition:fade`, `in:fly`, `out:slide` — enter/exit animations
- **Scoped Styles**: `<style>` block is scoped by default (no `scoped` attribute needed)
- **Svelte 5 Runes**: `$state()`, `$derived()`, `$effect()`, `$props()` — new reactivity model

## Svelte 5 Runes

- `$state(value)` — declares reactive state; replaces `let` reactive variables
- `$derived(expression)` — declares derived reactive value; replaces `$:` reactive declarations
- `$effect(() => { ... })` — declares side effect; replaces `$:` reactive statements and lifecycle hooks
- `$props()` — declares component props; replaces `export let`
- `$bindable(value)` — declares a bindable prop (two-way binding)
- `$inspect()` — debug utility for reactive values
- `*.svelte.ts` files can use runes outside of `.svelte` components

## Notable File Patterns

- `*.svelte` — Svelte components
- `+layout.svelte`, `+page.svelte` — SvelteKit route components
- `+error.svelte` — SvelteKit error page
- `*.svelte.ts` — Svelte 5 module with runes
- `svelte.config.js` — Svelte/SvelteKit configuration

## Edge Patterns

- Svelte components are `contains` by the parent that imports them
- `export let` declarations are `exports` edges (component props)
- `$storeName` references are `depends_on` edges to the store module
- `$: reactive = expr` declarations are `depends_on` on the referenced variables
- `import Component from './Component.svelte'` creates `imports` edges
- `<Component />` in template creates `contains` edge from parent to child
- `setContext`/`getContext` creates implicit `depends_on` across component tree
- Scoped styles are `related` to the component they style
- `$state()`, `$derived()`, `$effect()` in Svelte 5 are `depends_on` on their dependencies

## Summary Style

> "Svelte component implementing a counter with reactive doubled value and click handler."
> "SvelteKit page component fetching data from API with loading/error states."
> "Svelte store module exporting a writable store for user authentication state."
> "Svelte 5 component using runes for reactive form state and derived validation."
```

## 9. 与 Vue SFC 的对比

| 维度 | Vue SFC (VueSfcPlugin) | Svelte (SveltePlugin) |
|---|---|---|
| 编译器 | `@vue/compiler-sfc` | `svelte/compiler` |
| script 提取 | `descriptor.scriptSetup ?? descriptor.script` | `ast.instance ?? ast.module` |
| 行号来源 | `loc.start.line`（直接行号） | `start`（字符偏移，需转换） |
| TypeScript 检测 | `scriptBlock.lang === "ts"` | 正则检测 `<script lang="ts">` |
| template 组件 | PascalCase 标签 | 大写开头标签 + Component 节点 |
| 响应式语法 | `ref()`, `reactive()` 函数调用 | `$:` 标签语句 |
| Store 语法 | `import { useStore } from 'vuex/pinia'` | `$storeName` 自动订阅 |
| Props 声明 | `defineProps<T>()` 编译宏 | `export let x` |
| 样式作用域 | `<style scoped>` 显式声明 | `<style>` 默认 scoped |
| scan-project.mjs 修改 | 需更新语言 ID (`vue` → `vue-sfc`) | 不需要（ID 已一致） |
| 框架配置联动 | 需更新 Vue 框架的 languages | 不需要（无现有 Svelte 框架配置） |

## 10. 与 v3.0 的关键差异

| 维度 | v3.0（无 Tree-sitter 模式） | v4.0（独立 Plugin + 官方编译器） |
|---|---|---|
| 结构化分析 | 纯 LLM，非确定性 | svelte/compiler + TS tree-sitter，确定性 |
| 新增依赖 | 无 | `svelte` |
| 新增文件 | 2（配置 + 指南） | 3（配置 + 插件 + 指南） |
| 修改文件 | 2 | 4 |
| 指纹系统 | 降级模式（hasStructuralAnalysis: false） | 结构化指纹（hasStructuralAnalysis: true） |
| 行号精度 | LLM 估算 | 精确到行（编译器提供） |
| template 组件引用 | LLM 识别 | 编译器 AST 提取，确定性 |
| call graph | 不支持 | 支持（TS tree-sitter 提取 + 行号偏移） |
