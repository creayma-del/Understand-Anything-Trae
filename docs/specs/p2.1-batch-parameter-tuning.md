# P2.1: 批次参数调优与并发上限提升

> 版本: 1.1
> 日期: 2026-06-08
> 状态: 已评审（含评审修订）
> 前置: P1（前端生态深化）已完成
> 变更: 调优 `compute-batches.mjs` 硬编码参数（优化 C），提升 SKILL.md 并发上限（优化 G），减少 LLM 调用次数约 40%

## 1. 目标与范围

### 1.1 问题

`compute-batches.mjs` 的批次参数过于保守，导致 300+ 文件项目产生过多批次（~20 个），每个批次对应一次 LLM 子代理调度。同时 SKILL.md 规定的并发上限仅 5，进一步拉长了整体分析耗时。

具体表现：
- `MAX_COMMUNITY_SIZE = 35`：现代 LLM 上下文窗口 128k+，35 文件元数据仅 ~8-15k token，远未填满
- `MIN_BATCH_SIZE = 3`：过小批次（1-2 文件）的子代理启动 + prompt 构造开销 ~5-10s/次，与实际分析时间不成比例
- `MAX_MERGE_TARGET = 25`：合并后 misc 批次偏小，总批次数偏高
- `count-fallback batchSize = 12`：回退模式批次也偏小
- `MAX_E = 20`：非代码文件元数据更小（无 functions/classes/callGraph），20 文件上限过于保守
- 并发上限 5：Trae IDE 可支持更高并发

### 1.2 目标

- 300 文件项目批次数从 ~20 降至 ~12（减少 40% LLM 调用）
- 并发调度轮次从 ceil(20/5) = 4 轮降至 ceil(12/8) = 2 轮
- 分析精度不损失：Louvain 社区检测的语义分批优势在 60 文件以内仍然有效

### 1.3 范围

**包含**：
- `compute-batches.mjs` 中 5 个硬编码参数的调优
- `SKILL.md` 中并发上限从 5 提升至 8
- 对应测试用例更新

**不包含**：
- 其他 P2 优化项（A/B/D/E/F，各自独立规格）
- `compute-batches.mjs` 的架构变更（如读取 `exports-map.json`，属于优化 B）
- LLM prompt 优化
- 增量更新路径变更

## 2. 代码模式分析 / 背景分析

### 2.1 compute-batches.mjs 参数分布

`compute-batches.mjs` 中有 6 个硬编码参数控制批次大小和合并行为：

| 参数 | 位置 | 当前值 | 作用 |
|------|------|--------|------|
| `MAX_COMMUNITY_SIZE` | 第 369 行 | 35 | Louvain 社区最大文件数，超出则按字母序拆分 |
| `MIN_BATCH_SIZE` | 第 249 行 | 3 | 低于此值的 mergeable 批次被合并到 misc 批次 |
| `MAX_MERGE_TARGET` | 第 252 行 | 25 | 合并后 misc 批次的最大文件数 |
| `count-fallback batchSize` | 第 222 行 | 12 | Louvain 失败时回退到按字母序分批的每批文件数 |
| `MAX_E` | 第 173 行 | 20 | 非 code 文件 Group E 按父目录分批的最大文件数 |
| `MAX_NEIGHBORS` | 第 448 行 | 50 | neighborMap 中每个文件的跨批邻居上限 |

### 2.2 参数间依赖关系

```
MAX_COMMUNITY_SIZE (社区拆分上限)
  └→ 决定 Louvain 输出的最大批次大小
     └→ MAX_MERGE_TARGET 应 < MAX_COMMUNITY_SIZE
        （misc 批次不应超过社区拆分上限，否则拆分逻辑被绕过）

MIN_BATCH_SIZE (小批次合并阈值)
  └→ 低于此值的 mergeable 批次进入 misc 池
     └→ MAX_MERGE_TARGET 控制 misc 池中每个输出批次的大小

count-fallback batchSize (回退批次大小)
  └→ 独立参数，仅在 Louvain 失败时生效

MAX_E (非 code Group E 上限)
  └→ 独立参数，仅影响非代码文件分批

MAX_NEIGHBORS (跨批邻居上限)
  └→ 独立参数，控制 neighborMap 大小，不影响批次数
```

### 2.3 现有测试对参数的硬依赖

现有测试 `test_compute_batches.test.mjs` 中以下用例对参数值有硬依赖：

| 测试用例 | 硬依赖参数 | 硬依赖值 |
|---------|-----------|---------|
| `size enforcement — splits a 40-node clique into batches ≤ 35` | `MAX_COMMUNITY_SIZE` | 35 |
| `Group E MAX_E split — splits 25 .md files under docs/ into [20, 5]` | `MAX_E` | 20 |
| `merge-small — merges 100 isolated singletons into a small number of misc batches` | `MAX_MERGE_TARGET` | 25 |
| `fallback — falls back to count-based when Louvain throws` | `count-fallback batchSize` | 12 |

参数变更后，这些测试的预期值必须同步更新。

### 2.4 Token 预算分析

每个 file-analyzer 子代理的 prompt 包含：
- 系统指令 + 模板：~2k token
- 每个文件的元数据（path, language, sizeLines, fileCategory, imports, exports）：~200-500 token
- 每个文件的 neighborMap 条目：~100-300 token
- LLM 输出预算：~5k token/文件

60 文件批次的预估 token 消耗：
- 输入：2k + 60 × (400 + 200) = 38k token
- 输出：60 × 5k = 300k token（但实际远低于此，LLM 输出通常 ~1-2k token/文件）
- 实际输出估算：60 × 1.5k = 90k token
- **总计：~128k token**，在 128k 上下文窗口内

40 文件 misc 批次（非代码文件元数据更小）：
- 输入：2k + 40 × (200 + 100) = 14k token
- **总计：~74k token**，远在窗口内

## 3. 参数变更设计

### 3.1 变更总表

| 参数 | 当前值 | 新值 | 变更幅度 | 理由 |
|------|--------|------|---------|------|
| `MAX_COMMUNITY_SIZE` | 35 | 60 | +71% | 现代 LLM 128k+ 上下文窗口；60 文件元数据 ~15-30k token，远未填满 |
| `MIN_BATCH_SIZE` | 3 | 5 | +67% | 减少过小批次的调度开销（子代理启动 + prompt 构造 ~5-10s/次） |
| `MAX_MERGE_TARGET` | 25 | 40 | +60% | 合并后批次更大，减少总批次数；仍 < MAX_COMMUNITY_SIZE=60 |
| `count-fallback batchSize` | 12 | 25 | +108% | 回退模式也用更大批次，减少批次数 |
| `MAX_E` | 20 | 35 | +75% | 非代码文件元数据更小（无 functions/classes/callGraph），可容纳更多 |
| `MAX_NEIGHBORS` | 50 | 50 | 不变 | 已足够覆盖跨批上下文 |
| 并发上限 | 5 | 8 | +60% | Trae IDE 可支持更高并发 |

### 3.2 参数约束验证

| 约束 | 验证 |
|------|------|
| `MAX_MERGE_TARGET < MAX_COMMUNITY_SIZE` | 40 < 60 ✓ |
| `MIN_BATCH_SIZE < MAX_MERGE_TARGET` | 5 < 40 ✓ |
| `count-fallback batchSize ≤ MAX_COMMUNITY_SIZE` | 25 ≤ 60 ✓ |
| `MAX_E ≤ MAX_COMMUNITY_SIZE` | 35 ≤ 60 ✓ |
| `MAX_NEIGHBORS` 不变 | 50 ✓ |

### 3.3 效果预估（300 文件项目）

| 指标 | 优化前 | 优化后 | 改善 |
|------|--------|--------|------|
| Louvain 产出批次数 | ~20 | ~12 | -40% |
| LLM 调用次数 | 20 | 12 | -40% |
| 并发调度轮次 | ceil(20/5) = 4 | ceil(12/8) = 2 | -50% |
| 单批次最大 token | ~45k | ~128k | 在窗口内 |

  writeFileSync(
    join(root, '.understand-anything-trae', 'intermediate', 'scan-result.json'),
    JSON.stringify(scan));

  const result = runScript(root);
  expect(result.status).toBe(0);

  const batches = readBatches(root);
  // 40 config files under config/ → Group E with MAX_E=35 → [35, 5]
  const configBatches = batches.batches.filter(b =>
    b.files.every(f => f.path.startsWith('config/')));
  expect(configBatches.length).toBe(2);
  const sizes = configBatches.map(b => b.files.length).sort((a, b) => b - a);

});
