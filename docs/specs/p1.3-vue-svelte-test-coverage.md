# P1.3: Vue/Svelte 解析器测试覆盖

> 版本: 1.0
> 日期: 2026-06-05
> 状态: 已评审（修复 HIGH 问题后）
> 变更: 初始版本 — 为 VueSfcPlugin 和 SveltePlugin 建立完整的单元测试覆盖，填补当前测试缺口

## 1. 目标与范围

为 `VueSfcPlugin`（`packages/core/src/plugins/parsers/vue-sfc-parser.ts`）和 `SveltePlugin`（`packages/core/src/plugins/parsers/svelte-parser.ts`）建立完整的单元测试覆盖，确保：

- 所有公开方法（`analyzeFile`、`resolveImports`、`extractCallGraph`）的输入/输出行为正确
- 内部逻辑分支（降级守卫、行号偏移、template 组件提取、去重）均有测试覆盖
- 边界场景（空文件、无 script 块、TypeScript 检测）不遗漏
- 测试可独立运行，不依赖真实 TreeSitterPlugin 实例（通过 Mock 隔离）

**范围**：

| 包含 | 不包含 |
|---|---|
| VueSfcPlugin 单元测试 | TreeSitterPlugin 自身测试（已有覆盖） |
| SveltePlugin 单元测试 | `@vue/compiler-sfc` / `svelte/compiler` 内部行为测试 |
| TreeSitterPlugin Mock 设计 | 集成测试（需真实 WASM 加载） |
| 测试 fixture 文件设计 | LanguageConfig 测试（已在 `language-registry.test.ts` 覆盖） |
| `registerAllParsers` 回归验证 | LLM 分析流程测试 |

## 2. 当前测试覆盖缺口分析

### 2.1 现有测试现状

| 文件 | 覆盖情况 |
|---|---|
| `packages/core/src/__tests__/parsers.test.ts` | 覆盖 MarkdownParser、YAMLConfigParser、JSONConfigParser、TOMLParser、EnvParser、DockerfileParser、SQLParser、GraphQLParser、MakefileParser、ShellParser — **无 Vue/Svelte** |

| `packages/core/src/plugins/parsers/__tests__/` | **目录不存在**，Vue/Svelte 解析器零测试覆盖 |

### 2.2 关键风险路径（无测试覆盖）

| 风险路径 | 影响范围 | 严重程度 |
|---|---|---|
| `VueSfcPlugin.analyzeFile` 行号偏移计算 | 函数/类/导入/导出的行号全部错误 | Critical |
| `SveltePlugin.analyzeFile` 字符偏移→行号转换 | Svelte 场景下行号全部错误 | Critical |
| 降级守卫（tsPlugin 为 null） | 插件未初始化时运行时崩溃 | High |
| template 组件引用提取 | 组件依赖关系丢失 | High |
| script 中已 import 的组件与 template 引用去重 | imports 列表出现重复条目 | Medium |
| Svelte 双 script 块合并 | instance + module script 分析结果丢失 | High |
| TypeScript 检测（Svelte regex 方式） | TS 代码被当作 JS 解析，类型信息丢失 | Medium |
| `defineProps`/`defineEmits` 编译宏 | Vue 3 组件接口信息未提取 | Medium |

### 5.2 Fixture 设计原则

1. **最小化**: 每个 fixture 仅包含验证目标所需的最小内容，避免无关噪声
2. **确定性**: fixture 内容固定，不依赖外部状态或随机值
3. **行号可控**: 行号偏移相关 fixture 的换行位置必须精确可控，使用 `\n` 显式换行
4. **内联优先**: 简短 fixture（< 10 行）可直接在测试代码中内联定义，无需独立文件；复杂 fixture 使用独立文件
5. **共享复用**: 同一 fixture 可被多个测试用例复用（如 `basic-script-setup.vue` 被 VUE-001 和 VUE-009 共用）

## 6. Mock 策略

### 6.1 TreeSitterPlugin Mock 设计

VueSfcPlugin 和 SveltePlugin 均依赖 TreeSitterPlugin 进行 script 内容的结构化分析。测试中需 Mock TreeSitterPlugin，隔离其 WASM 加载和解析逻辑。

```typescript
import { vi } from "vitest";
import type { TreeSitterPlugin } from "../../plugins/tree-sitter-plugin.js";
import type { StructuralAnalysis, CallGraphEntry } from "../../types.js";

/**
 * 创建 TreeSitterPlugin 的 Mock 实例。
 *
 * @param analyzeResult analyzeFile 方法的返回值
 * @param callGraphResult extractCallGraph 方法的返回值
 */
function createMockTsPlugin(
  analyzeResult?: StructuralAnalysis,
  callGraphResult?: CallGraphEntry[],
): TreeSitterPlugin {
  return {
    name: "tree-sitter",
    languages: ["typescript", "javascript"],
    analyzeFile: vi.fn().mockReturnValue(analyzeResult ?? {
      functions: [],
      classes: [],
      imports: [],
      exports: [],
    }),
    extractCallGraph: vi.fn().mockReturnValue(callGraphResult ?? []),
  } as unknown as TreeSitterPlugin;
}
```

### 6.2 Mock 使用模式

```typescript
// 模式 1: 基础 Mock — 返回空分析
const emptyMock = createMockTsPlugin();

// 模式 2: 自定义返回值 — 验证 script 分析逻辑
const customMock = createMockTsPlugin({
  functions: [{ name: "greet", lineRange: [1, 1], params: [] }],
  classes: [],
  imports: [{ source: "vue", specifiers: ["ref"], lineNumber: 1 }],
  exports: [],
});

// 模式 3: 验证调用参数 — 确认 virtual.ts/virtual.js 路径
const spyMock = createMockTsPlugin();
// ... 调用 plugin.analyzeFile() ...
expect(spyMock.analyzeFile).toHaveBeenCalledWith("virtual.ts", expect.any(String));

// 模式 4: 不注入 Mock — 测试降级守卫
const plugin = new VueSfcPlugin(); // 不调用 init()
```

### 6.3 Mock 边界注意事项

| 注意事项 | 说明 |
|---|---|
| 不 Mock `@vue/compiler-sfc` | Vue 编译器是确定性纯 JS 库，直接使用真实实现 |
| 不 Mock `svelte/compiler` | Svelte 编译器同理，直接使用真实实现 |
| Mock 返回值行号从 1 开始 | Mock 的 `analyzeFile` 返回值中行号应从 script 内容的第 1 行开始，偏移修正由被测插件负责 |
| `extractCallGraph` Mock | 仅在测试 `extractCallGraph` 方法时需要提供，`analyzeFile` 测试中可使用默认空数组 |

## 8. 风险评估

| 风险 | 可能性 | 影响 | 缓解措施 |
|---|---|---|---|
| `@vue/compiler-sfc` 版本差异导致 fixture 解析结果不同 | 低 | 中 | 锁定 `@vue/compiler-sfc` 版本为 `^3.5.0`，CI 中验证 |
| `svelte/compiler` 版本差异（Svelte 4 vs 5 AST 结构不同） | 中 | 高 | fixture 和测试逻辑需兼容 Svelte 5 AST；如需支持 Svelte 4，需额外 fixture |
| Mock 返回值与真实 TreeSitterPlugin 行为不一致 | 中 | 中 | Mock 返回值需严格匹配 `StructuralAnalysis` 类型定义；后续集成测试验证 |
| Svelte 字符偏移→行号转换在多字节字符场景下不准确 | 低 | 中 | fixture 使用纯 ASCII 内容；多字节场景作为后续增强 |
| `defineProps`/`defineEmits` 作为编译宏被 `@vue/compiler-sfc` 转换后，tsPlugin 看到的代码与源码不同 | 中 | 低 | 测试关注最终提取结果是否正确，不依赖中间代码形式 |
| 测试 fixture 中行号偏移计算与实际不一致 | 低 | 高 | 行号偏移 fixture 使用精确换行控制，测试中硬编码预期行号并注释计算过程 |
