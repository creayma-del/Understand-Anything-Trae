> 版本: 1.1
> 日期: 2026-06-08
> 状态: 已评审（含评审修订）
> 前置: P2（性能优化）已规划

## 1. 目标与范围

### 1.1 问题

当前 `/understand` 管线中，imports 边的生成存在以下问题：

1. **LLM 丢失率高**：`file-analyzer.md` 指示 LLM 从 `batchImportData` 生成 imports 边，但实际 LLM 丢失约 25% 的 imports 边
2. **恢复逻辑是补丁**：`merge-batch-graphs.py` 的 `recover_imports_from_scan()` 仅在事后补充遗漏，而非主流程
3. **职责错位**：imports 边本质上是确定性数据（来自 project-scanner 的 importMap），不应依赖 LLM 生成

### 1.2 目标

- imports 边生成从 LLM 负责改为 `merge-batch-graphs.py` 确定性生成
- imports 边覆盖率从 ~75% 提升到 100%
- 消除 LLM 生成 imports 边的 token 开销
- 消除后处理恢复逻辑，改为确定性主流程

### 1.3 范围

**包含**：
- `merge-batch-graphs.py`：`recover_imports_from_scan()` → `generate_imports_from_scan()` 重构
- `file-analyzer.md`：移除 imports 边创建指令
- `SKILL.md`：Phase 2 dispatch prompt 调整
- 测试用例新增

**不包含**：
- 其他优化项（A/B/C/D/E/G，见 P2 总规格）
- `importMap` 本身的生成逻辑变更
- 增量更新路径变更
- Dashboard 展示逻辑变更

### 1.4 约束

- 所有变更必须保持分析结果与变更前等价（或更优）
- 不引入新的外部依赖
- 保持增量更新路径完全兼容
- LLM 若仍生成 imports 边（旧版 agent prompt 缓存等原因），合并脚本的去重逻辑必须正确处理

## 2. 背景分析

### 2.1 当前数据流

```
Phase 1 (project-scanner)
  → scan-result.json (含 importMap: {filePath → [resolvedPaths]})

Phase 2 (file-analyzer × N)
  → batchImportData 注入到 dispatch prompt
  → LLM 尝试从 batchImportData 生成 imports 边（丢失 ~25%）
  → batch-*.json 输出

Phase 2 末尾 (merge-batch-graphs.py)
  → merge_and_normalize() 合并批次
  → recover_imports_from_scan() 补充 LLM 遗漏的 imports 边
  → assembled-graph.json
```

### 2.2 丢失根因

LLM 丢失 imports 边的原因包括：

1. **枚举遗漏**：LLM 在遍历 `batchImportData[file]` 数组时跳过部分条目
2. **跨批次目标跳过**：LLM 误以为 imports 边目标必须在同批次内
3. **自我审查不完整**：自检指令要求 `imports` 边数 = `batchImportData[file].length`，但 LLM 自检经常通过而实际不满足
4. **批次构造丢失**：编排端构造 `batchImportData` 时可能丢失部分条目

### 2.3 importMap 的权威性

`scan-result.json` 中的 `importMap` 由 `extract-import-map.mjs` 确定性生成：

- 使用 tree-sitter 解析源码中的 import/require 语句
- 通过路径解析算法将相对/别名路径解析为项目内绝对路径
- 已过滤外部包（node_modules 等）
- 是 imports 关系的唯一权威数据源

因此，直接从 importMap 生成 imports 边是确定性、完整且准确的。

## 3. 接口设计

### 3.1 `generate_imports_from_scan()` 函数签名

```python
def generate_imports_from_scan(
    assembled: dict[str, Any],
    scan_result_path: Path,
) -> list[dict[str, Any]]:
    """从 importMap 确定性生成全部 imports 边。

    Args:
        assembled: assembled graph 字典，包含 "nodes" 和 "edges" 键
        scan_result_path: scan-result.json 文件路径（Path 对象）

    Returns:
        新增的 imports 边列表（已去重，不包含 assembled["edges"] 中已有的）
    """
```

### 3.2 返回值结构

每条新增 imports 边的格式：

```python
{
    "source": "file:src/index.ts",     # 节点 ID（非文件路径）
    "target": "file:src/utils.ts",     # 节点 ID（非文件路径）
    "type": "imports",
    "direction": "forward",
    "weight": 0.7,
}
```

**与现有 `recover_imports_from_scan()` 的差异**：

| 属性 | `recover_imports_from_scan()` | `generate_imports_from_scan()` |
|------|------------------------------|-------------------------------|
| 定位 | 补充遗漏 | 主流程生成 |
| 输入 | `assembled` dict + `scan_result_path` | `assembled` dict + `scan_result_path`（对齐） |
| 输出 | `(count, report_lines)` | `list[dict]`（边列表） |
| 去重标记 | `recoveredFromImportMap: True` | 无额外标记 |
| 调用方式 | 读取文件系统 | 读取文件系统 |

## 4. 实现细节

### 4.1 `merge-batch-graphs.py` 变更

#### 4.1.1 新增 `generate_imports_from_scan()` 函数

替换现有 `recover_imports_from_scan()` 函数（第 781-866 行）：

```python
def generate_imports_from_scan(
    assembled: dict[str, Any],
    scan_result_path: Path,
) -> list[dict[str, Any]]:
    """从 importMap 确定性生成全部 imports 边。

    遍历 importMap 中的每个 (source_path, [target_paths]) 条目，
    查找对应节点 ID，生成 imports 边。跳过节点集中不存在的路径，
    避免产生悬空边引用。与 assembled["edges"] 中的已有 imports 边去重。
    """
    nodes = assembled.get("nodes", [])
    existing_edges = assembled.get("edges", [])

    # 读取 importMap
    import_map: dict[str, list[str]] = {}
    if scan_result_path.is_file():
        try:
            scan = json.loads(scan_result_path.read_text(encoding="utf-8"))
            if isinstance(scan.get("importMap"), dict):
                import_map = scan["importMap"]
        except (OSError, json.JSONDecodeError):
            pass

    # 构建 filePath → nodeId 映射（仅映射 file 节点，排除 function/class 子节点）
    file_path_to_id: dict[str, str] = {}
    for n in nodes:
        if n.get("type") == "file":
            fp = n.get("filePath")
            if fp:
                file_path_to_id[fp] = n["id"]

    # 从 importMap 生成全部候选 imports 边
    imports_edges: list[dict[str, Any]] = []
    for source_path, targets in import_map.items():
        source_id = file_path_to_id.get(source_path)
        if not source_id:
            continue
        if not isinstance(targets, list):
            continue
        for target_path in targets:
            if not isinstance(target_path, str) or not target_path:
                continue
            target_id = file_path_to_id.get(target_path)
            if not target_id:
                continue
            if source_id == target_id:
                continue
            imports_edges.append({
                "source": source_id,
                "target": target_id,
                "type": "imports",
                "direction": "forward",
                "weight": 0.7,
            })

    # 去重：移除 existing_edges 中已有的 imports 边
    existing_imports = {
        (e["source"], e["target"])
        for e in existing_edges
        if e.get("type") == "imports"
    }
    new_edges = [
        e for e in imports_edges
        if (e["source"], e["target"]) not in existing_imports
    ]

    return new_edges
```

## 5. 集成变更

### 5.1 数据流变更

```
变更前:
  file-analyzer → LLM 生成 imports 边（~75% 覆盖率）
  merge-batch-graphs.py → recover_imports_from_scan() 补充遗漏

变更后:
  file-analyzer → 不生成 imports 边
  merge-batch-graphs.py → generate_imports_from_scan() 确定性生成全部 imports 边
```

### 5.2 向后兼容

| 场景 | 处理方式 |
|------|---------|
| LLM 仍生成 imports 边（旧版 prompt 缓存） | `generate_imports_from_scan()` 的去重逻辑正确处理：已有边保留，仅补充缺失 |
| 增量更新路径 | `scan-result.json` 在增量更新中保留（Phase 7 清理规则已排除），importMap 可用 |
| 无 `scan-result.json` | `import_map` 为空字典，跳过生成，报告 "skipped — no importMap available" |
| `scan-result.json` 中无 `importMap` 字段 | 同上，跳过生成 |

### 5.3 报告输出变更

| 变更前 | 变更后 |
|--------|--------|
| `Imports edge recovery:` | `Imports edge generation (deterministic):` |
| `Recovered N imports edges from importMap` | `Generated N imports edges from importMap` |
| `Skipped N importMap source files with no file: node in graph` | 不再单独报告跳过计数（确定性生成是主流程，跳过无节点对应的路径是正常行为） |
| 边含 `recoveredFromImportMap: True` 标记 | 边无额外标记（与 LLM 生成的 imports 边格式一致） |

#### MBG-001：确定性 imports 边生成

**场景**：importMap 包含 A→B, A→C 两条导入关系

**输入**：
```python

    {"id": "file:src/a.ts", "type": "file", "filePath": "src/a.ts", ...},
    {"id": "file:src/b.ts", "type": "file", "filePath": "src/b.ts", ...},

**预期**：返回 2 条 imports 边
- `(file:src/a.ts, file:src/b.ts, imports, forward, 0.7)`
- `(file:src/a.ts, file:src/c.ts, imports, forward, 0.7)`

**场景**：LLM 已生成 A→B 的 imports 边，importMap 包含 A→B, A→C

nodes = [

## 9. 风险评估

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| LLM 仍生成 imports 边（旧版 agent prompt 缓存） | 低 | 低 | `generate_imports_from_scan()` 的去重逻辑正确处理：已有边保留，仅补充缺失。不会产生重复边 |
| `scan-result.json` 在增量更新中被删除 | 低 | 中 | Phase 7 清理规则已排除 `scan-result.json`（`-not -name 'scan-result.json'`），不会删除 |
| importMap 中包含已删除文件的路径 | 中 | 低 | 节点集中不存在对应节点时跳过，不产生悬空边。这是正确行为——已删除文件不应出现在图中 |
| 非文件节点（function/class）的 filePath 映射冲突 | 低 | 低 | 已通过 `type == "file"` 过滤解决：`file_path_to_id` 仅映射 file 类型节点，function/class 子节点即使具有相同 filePath 也不会被映射，确保 imports 边始终指向 file 节点（见 MBG-009） |
| 移除 LLM imports 边指令后 LLM 生成其他边类型时误用 imports | 低 | 低 | `merge-batch-graphs.py` 的 Step 6 去重逻辑按 `(source, target, type, direction)` 去重，不会误删其他类型边 |
| 动态 `import()` 表达式覆盖缺口 | 中 | 低 | `extract-import-map.mjs` 当前仅捕获静态 import/require 语句，不捕获动态 `import()` 表达式。移除 LLM imports 边生成后，动态 `import()` 的语义导入关系将丢失。这是可接受的，因为 importMap 是权威数据源，且动态 `import()` 提取可作为 `extract-import-map.mjs` 的未来增强 |
