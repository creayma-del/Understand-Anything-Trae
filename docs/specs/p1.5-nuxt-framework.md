# P1.5: Nuxt Framework Configuration 详细设计规格

> 版本: 1.0
> 日期: 2026-06-05
> 状态: 已评审（修复 HIGH 问题后）
> 变更: 新增 Nuxt 框架配置，作为 Vue 框架的特化版本，支持 Nuxt 项目结构分析、框架检测与层级映射

## 1. 目标与范围

为项目新增 Nuxt 框架配置（`FrameworkConfig`），使系统能够：
- 自动检测 Nuxt 项目（与 Vue 项目区分）
- 识别 Nuxt 特有的目录约定与文件角色
- 为 Nuxt 项目提供准确的层级映射（`layerHints`）
- 通过 Skill 文档（`nuxt.md`）指导 LLM 分析 Nuxt 架构模式

**范围限定**：
- 仅新增 `FrameworkConfig` 注册与 Skill 文档，不涉及语言配置（`LanguageConfig`）变更
- Nuxt 项目仍使用 `vue-sfc`、`typescript`、`javascript` 语言配置
- 不涉及 Tree-sitter 或编译器层面的变更

## 2. Nuxt 项目结构分析

### 2.1 Nuxt 与 Vue 的核心区别

| 维度 | Vue（纯 SPA） | Nuxt |
|---|---|---|
| 路由方式 | 手动配置 Vue Router | 基于文件系统的自动路由 |
| 服务端渲染 | 需额外配置 | 默认支持 SSR / 混合渲染 |
| 自动导入 | 需 unplugin 插件 | 内置 auto-imports + auto-components |
| 服务端代码 | 无 | 内置 `server/` 目录支持 API routes |
| 配置文件 | `vite.config.ts` / `vue.config.js` | `nuxt.config.ts` |
| 入口文件 | `src/main.ts` + `src/App.vue` | 无显式入口，由框架自动生成 |

### 2.2 Nuxt 目录约定

```
nuxt-project/
├── nuxt.config.ts          # 框架配置（核心检测文件）
├── app.vue                 # 根组件（替代 Vue 的 App.vue）
├── pages/                  # 基于文件的路由
│   ├── index.vue           # → /
│   ├── about.vue           # → /about
│   └── users/
│       ├── index.vue       # → /users
│       └── [id].vue        # → /users/:id（动态路由）
├── components/             # 自动导入的组件
├── composables/            # 自动导入的 composable 函数
├── layouts/                # 布局模板
│   └── default.vue         # 默认布局
├── server/
│   ├── api/                # API 路由（→ /api/*）
│   │   └── hello.ts        # → GET /api/hello
│   ├── routes/             # 服务端路由
│   ├── middleware/          # 服务端中间件
│   └── utils/              # 服务端工具函数
├── middleware/              # 路由中间件（客户端）
├── plugins/                # 插件（自动注册）
├── utils/                  # 工具函数（自动导入）
├── assets/                 # 静态资源（构建处理）
├── public/                 # 公共静态文件（原样服务）
└── error.vue               # 全局错误页面
```

### 2.3 文件角色对照

| 文件 / 模式 | 角色 | 与 Vue 的差异 |
|---|---|---|
| `nuxt.config.ts` | 框架配置 | Vue 使用 `vite.config.ts` |
| `app.vue` | 根组件 | Nuxt 特有，替代 `src/App.vue` |
| `pages/**/*.vue` | 路由页面 | Nuxt 自动路由，Vue 需手动配置 |
| `layouts/*.vue` | 布局模板 | Nuxt 内置，Vue 需手动实现 |
| `server/api/**` | API 路由 | Nuxt 独有，Vue 无此概念 |
| `middleware/*.ts` | 路由中间件 | Nuxt 自动注册，Vue 需手动挂载 |
| `plugins/*.ts` | 插件 | Nuxt 自动注册，Vue 需 `app.use()` |

## 3. Layer 1: FrameworkConfig 实现

### 3.1 文件: `packages/core/src/languages/frameworks/nuxt.ts`

```typescript
import type { FrameworkConfig } from "../types.js";

export const nuxtConfig = {
  id: "nuxt",
  displayName: "Nuxt",
  languages: ["typescript", "javascript", "vue-sfc"],
  detectionKeywords: [
    "\"nuxt\":",
    "\"nuxtjs\":",
    "nuxt3",
    "@nuxt/",
  ],
  manifestFiles: ["package.json", "nuxt.config.ts", "nuxt.config.js"],
  promptSnippetPath: "./frameworks/nuxt.md",
  entryPoints: [
    "app.vue",
    "pages/index.vue",
  ],
  layerHints: {
    pages: "ui",
    components: "ui",
    layouts: "ui",
    composables: "service",
    server: "api",
    "server/api": "api",
    "server/routes": "api",
    "server/middleware": "middleware",
    "server/utils": "service",
    middleware: "middleware",
    plugins: "config",
    utils: "utility",
  },
} satisfies FrameworkConfig;
```

### 3.2 配置字段说明

| 字段 | 值 | 设计理由 |
|---|---|---|
| `id` | `"nuxt"` | 唯一标识，与 Vue 的 `"vue"` 区分 |
| `displayName` | `"Nuxt"` | 用户可见名称 |
| `languages` | `["typescript", "javascript", "vue-sfc"]` | 与 Vue 一致，Nuxt 项目使用相同语言集 |
| `detectionKeywords` | `["\"nuxt\":", "\"nuxtjs\":", "nuxt3", "@nuxt/"]` | 详见第 4 节 |
| `manifestFiles` | `["package.json", "nuxt.config.ts", "nuxt.config.js"]` | `package.json` 用于关键词检测；`nuxt.config.ts`/`nuxt.config.js` 作为辅助检测依据，确认 Nuxt 项目身份 |
| `promptSnippetPath` | `"./frameworks/nuxt.md"` | 指向 Nuxt 专属 Skill 文档 |
| `entryPoints` | `["app.vue", "pages/index.vue"]` | Nuxt 无显式 main.ts，入口为 app.vue 或首页 |
| `layerHints` | 12 条映射 | 详见第 3.3 节 |

### 3.3 layerHints 设计

```typescript
layerHints: {
  // UI 层
  pages: "ui",              // 路由页面组件
  components: "ui",         // 可复用 UI 组件
  layouts: "ui",            // 布局模板

  // Service 层
  composables: "service",   // 组合式函数
  "server/utils": "service", // 服务端工具函数

  // API 层
  server: "api",            // server 目录整体
  "server/api": "api",      // API 路由处理器
  "server/routes": "api",   // 服务端路由

  // Middleware 层
  "server/middleware": "middleware", // 服务端中间件
  middleware: "middleware",          // 客户端路由中间件

  // Config 层
  plugins: "config",        // 自动注册的插件

  // Utility 层
  utils: "utility",         // 自动导入的工具函数
}
```

**与 Vue layerHints 的差异**：

| 目录 | Vue 映射 | Nuxt 映射 | 差异原因 |
|---|---|---|---|
| `pages/` | `ui` | `ui` | 一致 |
| `components/` | `ui` | `ui` | 一致 |
| `composables/` | `service` | `service` | 一致 |
| `store/` / `stores/` | `service` | — | Nuxt 推荐使用 Pinia 模块，无独立 stores 目录约定 |
| `router/` | `config` | — | Nuxt 无 router 目录（文件路由自动生成） |
| `plugins/` | `config` | `config` | 一致 |
| `layouts/` | — | `ui` | Nuxt 独有目录 |
| `server/` | — | `api` | Nuxt 独有目录 |
| `middleware/` | — | `middleware` | Nuxt 独有目录（客户端路由中间件） |
| `utils/` | — | `utility` | Nuxt 独有（自动导入） |

### 3.4 与 Vue 框架配置的检测优先级处理

**问题**：Vue 的 `detectionKeywords` 中包含 `"nuxt"`，导致 Nuxt 项目同时匹配 Vue 和 Nuxt 两个框架配置。

**解决方案**：在 `builtinFrameworkConfigs` 数组中，将 `nuxtConfig` 排在 `vueConfig` 之前，**并必须从 Vue 的 `detectionKeywords` 中移除 `"nuxt"`**，以避免检测冲突。

**理由**：`detectFrameworks()` 遍历 `this.byId.values()`（即注册顺序），一旦匹配即加入结果。Nuxt 是 Vue 的超集，Nuxt 项目应被识别为 Nuxt 而非 Vue。将 Nuxt 排在 Vue 前面可确保：
1. Nuxt 项目优先匹配 `nuxtConfig`（更精确）
2. 即使同时匹配，两个框架都会被检测到，但 Nuxt 排在前面，消费者可优先使用
3. 从 Vue 的 `detectionKeywords` 移除 `"nuxt"` 可避免 Nuxt 项目被误识别为纯 Vue 项目

> **重要**：Vue 的 `detectionKeywords` 必须移除 `"nuxt"` 关键词。如果保留，Nuxt 项目将同时被检测为 Vue 和 Nuxt，导致检测冲突。Nuxt 的检测应完全由 `nuxtConfig.detectionKeywords` 负责。

**Vue 配置变更**：

```typescript
// packages/core/src/languages/frameworks/vue.ts
// 变更前
detectionKeywords: ["vue", "@vue/cli-service", "nuxt", "vite-plugin-vue"],

// 变更后
detectionKeywords: ["vue", "@vue/cli-service", "vite-plugin-vue"],
```

## 4. Layer 2: 框架检测逻辑

### 4.1 detectionKeywords 设计

| 关键词 | 匹配目标 | 误检风险 |
|---|---|---|
| `"\"nuxt\":"` | package.json 中 `"nuxt": "^3.x"` 依赖声明 | 极低 — JSON key 精确匹配 |
| `"\"nuxtjs\":"` | 旧版 Nuxt 2 的 `"nuxtjs": "^2.x"` | 极低 — JSON key 精确匹配 |
| `"nuxt3"` | package.json 中 `"nuxt3"` 或 `nuxt3` 相关依赖 | 低 — 仅 Nuxt 3 项目使用 |
| `"@nuxt/"` | Nuxt 官方模块前缀（如 `@nuxt/content`、`@nuxt/image`） | 极低 — 仅 Nuxt 生态使用 |

**设计原则**：
- 优先使用 JSON key 精确匹配（`"\"nuxt\":"`），避免匹配到值中的 "nuxt" 字符串
- 覆盖 Nuxt 2（`nuxtjs`）和 Nuxt 3（`nuxt`、`nuxt3`）两个大版本
- `@nuxt/` 前缀可检测使用 Nuxt 官方模块的项目

### 4.2 manifestFiles 设计

```typescript
manifestFiles: ["package.json", "nuxt.config.ts", "nuxt.config.js"],
```

- **`package.json`**: 主检测文件，解析 dependencies + devDependencies 进行关键词匹配
- **`nuxt.config.ts` / `nuxt.config.js`**: 辅助检测文件，确认 Nuxt 项目身份。虽然不适合关键词匹配，但其存在可增强检测置信度

**设计原则**：
- `package.json` 是标准化的 JSON 格式，关键词匹配最可靠
- `nuxt.config.ts` / `nuxt.config.js` 是 Nuxt 项目特有的配置文件，其存在可作为辅助确认
- 与 SvelteKit 框架配置（使用 `svelte.config.js` / `svelte.config.ts`）保持一致的设计模式

### 4.3 检测流程示例

```
输入: package.json = { "dependencies": { "nuxt": "^3.12.0", "@nuxt/content": "^2.0.0" } }

1. 遍历 builtinFrameworkConfigs（顺序: nuxt → nextjs → react → vue → express）
2. nuxtConfig.manifestFiles = ["package.json"] → 匹配
3. nuxtConfig.detectionKeywords:
   - "\"nuxt\":" → package.json 中包含 "nuxt": → ✅ 匹配
4. 检测结果: [nuxtConfig]

注意: vueConfig.detectionKeywords 已移除 "nuxt"，因此不会误匹配
```

## 5. Layer 3: Skill 文档

### 5.1 文件: `skills/understand/frameworks/nuxt.md`

```markdown
# Nuxt Framework Addendum

> Injected into file-analyzer and architecture-analyzer prompts when Nuxt is detected.
> Do NOT use as a standalone prompt — always appended to the base prompt template.

## Nuxt Project Structure

When analyzing a Nuxt project, apply these additional conventions on top of the base analysis rules.

### Canonical File Roles

| File / Pattern | Role | Tags |
|---|---|---|
| `nuxt.config.ts` | Framework configuration — defines modules, runtime config, app behavior | `config`, `entry-point` |
| `app.vue` | Root application component — wraps NuxtPage and sets top-level layout | `entry-point`, `ui` |
| `pages/**/*.vue` | File-based route pages — path determines URL | `ui`, `routing` |
| `layouts/*.vue` | Layout templates — wrap page content with shared UI (nav, footer) | `ui`, `config` |
| `components/**/*.vue` | Auto-imported UI components — no manual import needed | `ui` |
| `composables/*.ts`, `composables/**/*.ts` | Auto-imported composable functions — reusable stateful logic | `service`, `utility` |
| `server/api/**` | API route handlers — define serverless endpoints (GET, POST, etc.) | `api-handler` |
| `server/routes/**` | Server routes — more flexible server-side routing | `api-handler` |
| `server/middleware/*.ts` | Server middleware — intercepts requests before reaching route handlers | `middleware` |
| `server/utils/*.ts` | Server utility functions — shared server-side logic | `service` |
| `middleware/*.ts` | Client-side route middleware — navigation guards (auth, redirects) | `middleware` |
| `plugins/*.ts`, `plugins/**/*.ts` | Auto-registered plugins — extend app functionality at startup | `config` |
| `utils/*.ts` | Auto-imported utility functions — pure helper functions | `utility` |
| `error.vue` | Global error page — catches unhandled errors across the app | `ui`, `config` |
| `assets/**` | Build-processed static assets — styles, images processed by Vite | `resource` |
| `public/**` | Public static files — served as-is at root path | `resource` |

### Edge Patterns to Look For

**Page-to-layout binding** — When a page component declares `definePageMeta({ layout: 'custom' })`, create `depends_on` edges from the page to the named layout. Pages without explicit layout assignment use `layouts/default.vue`.

**Component-to-composable** — When a component or page calls a `useX()` function from `composables/`, create `depends_on` edges from the consumer to the composable module. Nuxt auto-imports make these implicit — check for `use`-prefixed calls even without explicit import statements.

**Server route-to-server util** — When a `server/api/` handler imports from `server/utils/`, create `depends_on` edges. Server utilities are auto-imported within the server directory — look for direct function calls without import statements.

**Auto-imported component usage** — When a template uses a PascalCase component tag that matches a file in `components/`, create `contains` edges from the parent to the child component, even without explicit import statements. Nuxt resolves components by directory path and filename.

**Plugin registration** — Plugins in `plugins/` are auto-registered. When a plugin calls `defineNuxtPlugin()` and provides `provide` or `setup`, create `configures` edges from the plugin to the capabilities it injects.

**Middleware chain** — When `middleware/*.ts` files export route guard functions, and pages reference them via `definePageMeta({ middleware: ['auth'] })`, create `depends_on` edges from the page to the middleware module.

### Architectural Layers for Nuxt

Assign nodes to these layers when detected:

| Layer ID | Layer Name | What Goes Here |
|---|---|---|
| `layer:ui` | UI Layer | `pages/`, `components/`, `layouts/`, `app.vue`, `error.vue` |
| `layer:api` | API Layer | `server/api/`, `server/routes/` |
| `layer:service` | Service Layer | `composables/`, `server/utils/` |
| `layer:middleware` | Middleware Layer | `server/middleware/`, `middleware/` (client-side) |
| `layer:config` | Config Layer | `nuxt.config.ts`, `plugins/`, `app.vue` (as layout host) |

### Notable Patterns to Capture in languageLesson

- **Auto-imports**: Nuxt automatically imports components, composables, and utilities — no manual import statements needed. This means dependencies may be implicit and must be inferred from usage patterns (e.g., `useX()` calls imply a `composables/useX.ts` file).

- **File-based routing**: The `pages/` directory defines routes by file path — `pages/users/[id].vue` maps to `/users/:id`. Dynamic segments use `[param]` syntax. This replaces manual Vue Router configuration.

- **Server routes**: The `server/api/` directory creates API endpoints by file path — `server/api/hello.ts` maps to `/api/hello`. Each file exports an event handler using `defineEventHandler()`. This is Nuxt's built-in alternative to Express or other server frameworks.

- **Hybrid rendering**: Nuxt supports per-route rendering modes via `routeRules` in `nuxt.config.ts` — routes can be SSR, SPA, prerendered, or use SWR caching. Check `routeRules` to understand the rendering strategy for each page.

- **Layout system**: `layouts/default.vue` wraps all pages by default. Custom layouts are selected per-page via `definePageMeta({ layout: 'custom' })`. Layouts contain `<slot />` where page content is injected.

- **Nuxt modules**: `@nuxt/` prefixed packages (e.g., `@nuxt/content`, `@nuxt/image`) are Nuxt modules that extend the framework. They are registered in `nuxt.config.ts` modules array and may inject components, composables, or server routes.
```

### 7.3 测试代码

```typescript
// packages/core/src/__tests__/framework-registry.test.ts — 新增测试

describe("Nuxt framework detection", () => {
  it("detects Nuxt 3 from package.json", () => {
    const registry = FrameworkRegistry.createDefault();
    const detected = registry.detectFrameworks({
      "package.json": '{"dependencies": {"nuxt": "^3.12.0"}}',
    });
    expect(detected.some(f => f.id === "nuxt")).toBe(true);
  });

  it("detects Nuxt 2 from package.json", () => {
    const registry = FrameworkRegistry.createDefault();
    const detected = registry.detectFrameworks({
      "package.json": '{"dependencies": {"nuxtjs": "^2.16.0"}}',
    });
    expect(detected.some(f => f.id === "nuxt")).toBe(true);
  });

  it("detects Nuxt from @nuxt/ modules", () => {
    const registry = FrameworkRegistry.createDefault();
    const detected = registry.detectFrameworks({
      "package.json": '{"dependencies": {"@nuxt/content": "^2.0.0"}}',
    });
    expect(detected.some(f => f.id === "nuxt")).toBe(true);
  });

  it("Nuxt takes priority over Vue when both match", () => {
    const registry = FrameworkRegistry.createDefault();
    const detected = registry.detectFrameworks({
      "package.json": '{"dependencies": {"nuxt": "^3.12.0", "vue": "^3.4.0"}}',
    });
    const nuxtIndex = detected.findIndex(f => f.id === "nuxt");
    const vueIndex = detected.findIndex(f => f.id === "vue");
    expect(nuxtIndex).toBeLessThan(vueIndex);
  });
});

describe("Nuxt layerHints", () => {
  const config = nuxtConfig;

  it("maps pages to ui", () => {
    expect(config.layerHints?.["pages"]).toBe("ui");
  });

  it("maps server/api to api", () => {
    expect(config.layerHints?.["server/api"]).toBe("api");
  });

  it("maps composables to service", () => {
    expect(config.layerHints?.["composables"]).toBe("service");
  });

  it("maps middleware to middleware", () => {
    expect(config.layerHints?.["middleware"]).toBe("middleware");
  });
});
```

## 9. 风险评估

| 风险 | 等级 | 影响 | 缓解措施 |
|---|---|---|---|
| Vue detectionKeywords 移除 `"nuxt"` 导致现有 Nuxt 项目不再被识别为 Vue | 低 | Nuxt 项目应被识别为 Nuxt 而非 Vue，这是预期行为 | Nuxt 配置提供更精确的检测，Vue 移除 `"nuxt"` 是正确行为 |
| Nuxt 项目同时匹配 Nuxt 和 Vue 两个框架 | 低 | `detectFrameworks()` 返回两个结果，消费者需选择优先级 | Nuxt 排在 Vue 之前，消费者可取第一个结果 |
| `@nuxt/` 关键词误匹配非 Nuxt 项目 | 极低 | `@nuxt/` 是 Nuxt 官方模块命名空间，几乎不可能出现在非 Nuxt 项目中 | 可接受的风险 |
| Nuxt 2 与 Nuxt 3 项目结构差异 | 中 | Nuxt 2 使用 `layouts/`、`pages/` 但无 `server/` 目录，Skill 文档主要描述 Nuxt 3 | Skill 文档中标注 Nuxt 2 差异，检测关键词覆盖两个版本 |
| `builtinFrameworkConfigs` 顺序依赖 | 低 | 检测优先级依赖数组顺序，未来新增框架可能打乱顺序 | 在 `index.ts` 中添加注释说明顺序重要性 |
