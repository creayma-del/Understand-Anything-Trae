# P0.3: SCSS/Sass 增强设计规格

> 版本: 3.0
> 日期: 2026-06-03
> 状态: 已评审
> 变更: v3.0 评审后修订：Sass 语法描述修正、interpolation 拼写修正、filePatterns 语义明确、PostCSS 配置补充、测试补充

## 1. 目标

增强现有 CSS 语言配置对 SCSS/Sass 的支持，包括：
- 扩展 CSS 配置的 concepts 和 filePatterns
- 补充 SCSS/Sass 特有的 LLM 分析提示词
- 增加 `.sass` 扩展名支持
- 增加 Tailwind CSS 配置文件识别
- 增加 PostCSS 配置文件识别

### 2.3 scan-project.mjs 中的映射

**文件**: `skills/understand/scan-project.mjs`

`LANGUAGE_BY_EXT` 映射（第 127-130 行）：
```javascript
'.css': 'css',
'.scss': 'css',
'.sass': 'css',
'.less': 'css',
```

`.sass` **已在** `LANGUAGE_BY_EXT` 中映射为 `'css'`，与 core 包的 `cssConfig.id` 一致。

`CATEGORY_BY_EXT` 映射（第 280-285 行）：
```javascript
'.css': 'markup',
'.scss': 'markup',
'.sass': 'markup',
'.less': 'markup',
```

**关键发现**: CSS/SCSS/Sass 文件被分类为 `'markup'` 而非 `'code'`。这意味着：
1. `extract-structure.mjs` 不会对 CSS 文件执行调用图提取（仅 `code`/`script` 类别才提取）
2. CSS 文件在 `extract-import-map.mjs` 中被跳过（仅 `code` 类别才解析导入）
3. CSS 文件在 `extract-structure.mjs` 中仍会通过 `PluginRegistry` 执行结构分析（如果有对应的 parser）

### 2.4 缺失项

| 缺失项 | 影响 | 优先级 |
|---|---|---|
| `.sass` 扩展名未包含在 core 包配置中 | `.sass` 缩进语法文件不被 `LanguageRegistry` 识别 | 高 |
| SCSS/Sass 特有 concepts 未包含 | LLM 分析缺少 SCSS 特征理解 | 高 |
| filePatterns.config 为空 | Tailwind/PostCSS 配置文件不被关联 | 中 |
| filePatterns.entryPoints 为空 | 全局样式入口不被识别 | 低 |
| CSS-in-JS 模式未覆盖 | styled-components/emotion 不被识别 | 低（P1 范围） |

### 3.4 filePatterns 匹配语义

`filePatterns` 中的字符串匹配语义如下（与项目中其他语言配置一致）：
- **entryPoints**: 精确文件名匹配（如 `"globals.css"` 匹配任意目录下的 `globals.css`）
- **barrels**: 精确文件名匹配
- **tests**: glob 模式匹配
- **config**: 精确文件名匹配

**跨语言域说明**: `filePatterns.config` 中的 JS/TS 文件（如 `tailwind.config.js`）是跨语言域的配置关联——CSS 语言消费这些配置，但它们由 JS/TS 语言定义。`LanguageRegistry.getByExtension('.js')` 仍返回 `javascriptConfig`，不受 CSS filePatterns 影响。下游消费方不应假设 config 文件与当前语言扩展名一致。

## 4. Layer 3: LLM 指南增强

### 4.1 修改: `skills/understand/languages/css.md`

在现有内容基础上增加以下章节：

```markdown
## SCSS/Sass Specifics

### Module System
- `@use 'module'` — loads a module and makes its members available as `module.$var` or `module.mixin()`
- `@use 'module' as *` — loads a module and makes its members available without namespace
- `@forward 'module'` — re-exports a module's members from the current file
- `@use` replaces the deprecated `@import` (which is global and causes duplication)

### Partial Files
- Files starting with `_` (e.g., `_variables.scss`) are partials — not compiled to CSS directly
- Partials are loaded via `@use 'variables'` (omit the `_` prefix and extension)
- Common partials: `_variables.scss`, `_mixins.scss`, `_functions.scss`, `_reset.scss`

### Built-in Functions
- `lighten()`, `darken()`, `adjust-hue()` — color manipulation
- `map-get()`, `map-merge()` — map operations
- `str-index()`, `str-insert()` — string operations
- `math.div()`, `math.round()` — math operations (Sass 1.33+)

### Sass Indented Syntax (.sass)
- Uses indentation instead of braces and semicolons
- Properties are declared as `property: value` (colon suffix, no semicolons). Note: The older `:property value` (colon prefix) syntax is deprecated.
- Same features as SCSS but different syntax

## Tailwind CSS Specifics

### Configuration
- `tailwind.config.js/ts` — defines content paths, theme extensions, plugins
- `@tailwind base;` / `@tailwind components;` / `@tailwind utilities;` — CSS entry directives
- `content: ['./src/**/*.{html,js,ts,jsx,tsx,vue,svelte}']` — purge configuration

### Utility Patterns
- `@apply` — inlines utility classes in custom CSS
- `theme()` — references theme values in custom CSS: `color: theme('colors.primary')`
- `@layer` — Tailwind uses cascade layers for base/components/utilities ordering
- Arbitrary values: `w-[100px]`, `text-[#1da1f2]`, `grid-cols-[repeat(4,1fr)]`

### Edge Patterns for Tailwind
- `tailwind.config.js` is `configures` all CSS files that use `@tailwind` directives
- Component files using utility classes are `related` to `tailwind.config.js`
- `@apply` usage creates `depends_on` edges to the Tailwind utility layer

## PostCSS Specifics

### Configuration
- `postcss.config.js/ts` — defines PostCSS plugins and their options
- `.postcssrc.js` / `.postcssrc.json` / `.postcssrc.yml` — alternative config formats
- Common plugins: `autoprefixer`, `postcss-preset-env`, `cssnano`, `tailwindcss`

### Edge Patterns for PostCSS
- `postcss.config.js` is `configures` all CSS files processed by the build pipeline
- PostCSS plugins transform CSS at build time — not visible in source code
- `postcss.config.js` often `depends_on` `tailwind.config.js` when using Tailwind as a PostCSS plugin
```

## 5. 不涉及 Tree-sitter 提取器

CSS/SCSS/Sass 当前**没有** Tree-sitter 提取器，也不在 P0 范围内新增。原因：

1. CSS/SCSS 的结构化提取价值较低（无函数调用图、无导入解析需求）
2. CSS 的 `@import` 和 `@use` 已通过正则提取在 `extract-import-map.mjs` 中处理
3. LLM 分析已能覆盖 CSS/SCSS 的语义理解

如果未来需要 CSS Tree-sitter 提取器，可作为独立 P1 任务。

## 6. 导入解析 — 无需修改

### 6.1 CSS/SCSS 导入模式

CSS/SCSS 的导入不在 `extract-import-map.mjs` 的当前处理范围内（该脚本只处理 `fileCategory === 'code'` 的文件）。CSS/SCSS 文件被分类为 `'markup'`，`importMap[path] = []`。

如果需要支持 CSS/SCSS 导入解析，需要：

1. 将 CSS/SCSS 文件的 `fileCategory` 从 `'markup'` 改为 `'code'`（在 `scan-project.mjs` 的 `CATEGORY_BY_EXT` 中）
2. 或者在 `extract-import-map.mjs` 中增加 CSS/SCSS 的正则提取

**P0 决策**: 不修改 CSS/SCSS 的文件分类和导入解析。CSS/SCSS 的依赖关系由 LLM 分析处理。原因：
- 改变 `fileCategory` 会影响整个管线的文件分类逻辑
- CSS `@import`/`@use` 的解析需要单独实现，工作量较大
- LLM 分析已能识别 CSS 文件间的依赖关系

## 9. 影响范围

本项变更为**纯增量扩展**，不修改任何现有行为：

- `extensions` 新增 `.sass`，不影响已有 `.css/.scss/.less` 的匹配
- `concepts` 新增条目，不影响已有 concepts 的使用
- `filePatterns` 新增条目，不影响已有空数组的逻辑
- LLM 指南新增章节，不影响已有章节内容
- `scan-project.mjs` 无需修改（`.sass` 已映射）
- `fileCategory` 保持 `'markup'` 不变

## 10. 与 P0.1/P0.2 的关系

P0.3 是独立变更，与 P0.1（Vue SFC）和 P0.2（Svelte）无代码依赖。但存在以下语义关联：

| 关联点 | 说明 |
|---|---|
| Vue SFC `<style>` 块 | Vue SFC 中的 `<style lang="scss">` 使用 SCSS 语法，P0.3 的 concepts 增强有助于 LLM 理解 Vue 组件中的 SCSS |
| Svelte `<style>` 块 | Svelte 中的 `<style>` 默认 scoped，可使用 SCSS（需预处理器），P0.3 的 concepts 增强同样适用 |
| Tailwind + Vue/Svelte | Tailwind CSS 常与 Vue/Svelte 项目配合使用，P0.3 的 Tailwind 配置识别有助于框架检测 |
