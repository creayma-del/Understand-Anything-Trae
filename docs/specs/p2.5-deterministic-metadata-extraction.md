## 1. 目标与范围

### 1.1 目标

将 Phase 1 中 `project-scanner` 子代理的 Step A（LLM 调用）替换为确定性脚本 `extract-metadata.mjs`，消除一次冗余 LLM 调用（30-60 秒），同时提升 `name` 和 `frameworks` 字段的准确性。

### 1.2 问题分析

当前 `project-scanner.md` 的 Step A 使用 LLM 完成以下工作：

1. 读取 `package.json` → 提取 `name`、`description`、`dependencies`、`devDependencies`
2. 读取 `README.md` → 提取前 ~10 行作为 `readmeHead`
3. 从依赖列表匹配已知框架 → `frameworks[]`
4. 检测基础设施工具（Dockerfile、GitHub Actions、GitLab CI）→ 追加到 `frameworks[]`
5. 从 Step B 的 `byLanguage` 统计 → `languages[]`

这 5 项操作全部是**纯提取/匹配任务**，不需要 LLM 的语义理解能力。LLM 在此场景中反而存在以下问题：

- **耗时**：单次 LLM 调用 30-60 秒，占 Phase 1 总耗时的 50%+
- **不确定性**：相同输入可能产生不同输出（`name` 格式化差异、`frameworks` 顺序差异）
- **准确性风险**：LLM 可能"合成"不存在的 description、遗漏框架匹配、或错误推断

### 1.3 范围

**包含**：
- 新增 `extract-metadata.mjs` 确定性脚本
- 修改 `SKILL.md` Phase 1 流程：不再派遣 `project-scanner` 子代理
- 修改 `project-scanner.md` Step A：从 LLM 改为运行 `extract-metadata.mjs`
- 新增测试文件 `test_extract_metadata.test.mjs`

**不包含**：
- Phase 3 的 `assemble-reviewer` LLM 替换（优化 E，独立 spec）
- `scan-project.mjs`、`extract-import-map.mjs`、`analyze-project.mjs` 的修改
- 增量更新路径的修改
- 新增外部依赖

### 1.4 约束

- 输出 JSON 结构必须与 `project-scanner.md` Step A 的输出字段完全一致
- 框架匹配规则必须与 `project-scanner.md` Step A 的已知框架列表完全一致
- 不引入新的外部依赖
- 遵循现有脚本模式：CLI 参数、stderr 日志、JSON 输出

## 3. 详细设计

### 3.1 新增脚本: `extract-metadata.mjs`

**文件路径**: `understand-anything-trae-plugin/skills/understand/extract-metadata.mjs`

#### 3.1.1 CLI 接口

```bash
node <SKILL_DIR>/extract-metadata.mjs <projectRoot> <outputPath>
```

| 参数 | 类型 | 说明 |
|------|------|------|
| `projectRoot` | 绝对路径 | 项目根目录 |
| `outputPath` | 绝对路径 | 输出 JSON 文件路径 |

#### 3.1.2 处理流程

```
1. 读取 package.json → name, description, dependencies, devDependencies
2. 读取 README.md → 前 10 行作为 readmeHead
3. 合并 dependencies + devDependencies → 匹配 KNOWN_FRAMEWORKS → frameworks[]
4. 检测基础设施工具:
   - Dockerfile 存在 → 追加 "Docker"
   - .github/workflows/*.yml 存在 → 追加 "GitHub Actions"
   - .gitlab-ci.yml 存在 → 追加 "GitLab CI"
5. 读取 scan-result.json 的 stats.byLanguage → languages[]
6. 组装输出 JSON → 写入 outputPath
```

#### 3.1.3 输出 JSON 结构

```json
{
  "name": "package-name" | "<directory-name>",
  "description": "synthesized description" | "No description available",
  "rawDescription": "package description" | "",
  "readmeHead": "first 10 lines" | "",
  "frameworks": ["React", "Vite", ...],
  "languages": ["typescript", "css", "markdown", ...]
}
```

**字段说明**：

| 字段 | 类型 | 来源 | 回退策略 |
|------|------|------|---------|
| `name` | `string` | `package.json` `name` → 目录名 | 目录名（`path.basename(projectRoot)`） |
| `description` | `string` | `rawDescription` → `readmeHead` 首段合成 → "No description available" | "No description available" |
| `rawDescription` | `string` | `package.json` `description` | 空字符串 `""` |
| `readmeHead` | `string` | `README.md` 前 10 行 | 空字符串 `""` |
| `frameworks` | `string[]` | 依赖匹配 + 基础设施检测 | 空数组 `[]` |
| `languages` | `string[]` | `scan-result.json` 的 `stats.byLanguage` 键 | 空数组 `[]` |

#### 3.1.4 框架匹配规则

```javascript
const KNOWN_FRAMEWORKS = {
  'react': 'React',
  'vue': 'Vue',
  'svelte': 'Svelte',

  'express': 'Express',
  'fastify': 'Fastify',

  'redux': 'Redux',
  'zustand': 'Zustand',
  'mobx': 'MobX',

```

**匹配逻辑**：

1. 收集 `package.json` 的 `dependencies` 和 `devDependencies` 的所有键
2. 对每个键，检查 `KNOWN_FRAMEWORKS` 是否包含该键
3. 匹配成功则将对应值（友好名称）加入 `frameworks[]`
4. `frameworks[]` 按 `KNOWN_FRAMEWORKS` 的定义顺序排列（确定性排序）

**基础设施工具检测**：

| 检测条件 | 追加值 |
|---------|--------|
| `projectRoot/Dockerfile` 存在 或 `projectRoot/Dockerfile.*` 存在 | `"Docker"` |
| `projectRoot/.github/workflows/` 目录下存在 `*.yml` 或 `*.yaml` 文件 | `"GitHub Actions"` |
| `projectRoot/.gitlab-ci.yml` 存在 | `"GitLab CI"` |

基础设施检测结果追加到 `frameworks[]` 末尾，按 Docker → GitHub Actions → GitLab CI 顺序。

#### 3.1.5 README.md 读取

```javascript
function extractReadmeHead(projectRoot) {
  const readmeNames = ['README.md', 'README.rst', 'README', 'readme.md'];
  for (const name of readmeNames) {
    const filePath = path.join(projectRoot, name);
    if (existsSync(filePath)) {
      try {
        const content = readFileSync(filePath, 'utf-8');
        const lines = content.split('\n').slice(0, 10);
        return lines.join('\n');
      } catch {
        // 文件不可读，跳过
        process.stderr.write(`Warning: extract-metadata: ${name} exists but cannot be read — skipping\n`);
        return '';
      }
    }
  }
  return '';
}
```

#### 3.1.6 languages 字段来源

`languages` 从 `scan-result.json` 的输出文件中读取 `stats.byLanguage` 的键集合：

```javascript
function extractLanguages(scanResultPath) {
  try {
    const scanResult = JSON.parse(readFileSync(scanResultPath, 'utf-8'));
    const byLanguage = scanResult?.stats?.byLanguage;
    if (byLanguage && typeof byLanguage === 'object') {
      return Object.keys(byLanguage).sort();
    }
  } catch {
    process.stderr.write('Warning: extract-metadata: cannot read scan-result.json for languages — returning empty array\n');
  }
  return [];
}
```

**输入文件路径约定**：`scan-result.json` 位于 `<projectRoot>/.understand-anything-trae/intermediate/scan-result.json`（由 SKILL.md Phase 1 步骤 4 合并产出）。

#### 3.1.7 错误处理与 stderr 日志

遵循项目现有脚本模式：

| 场景 | 行为 | stderr 输出 |
|------|------|------------|
| `package.json` 不存在 | 跳过，`name`/`rawDescription` 从其他来源获取 | 无 |
| `package.json` 存在但 JSON 解析失败 | 跳过 | `Warning: extract-metadata: package.json exists but is not valid JSON — skipping` |
| `README.md` 不存在 | `readmeHead` 为空 | 无 |
| `README.md` 不可读 | `readmeHead` 为空 | `Warning: extract-metadata: README.md exists but cannot be read — skipping` |
| `scan-result.json` 不存在 | `languages` 为空数组 | `Warning: extract-metadata: cannot read scan-result.json for languages — returning empty array` |
| Dockerfile 检测失败（权限等） | 跳过 Docker 检测 | `Warning: extract-metadata: Dockerfile detection failed — <error message>` |
| `.github/workflows/` 读取失败 | 跳过 GitHub Actions 检测 | `Warning: extract-metadata: GitHub Actions detection failed — <error message>` |

脚本始终以 exit code 0 退出（除非 CLI 参数缺失）。部分失败不阻止输出。

#### 3.1.8 完整伪代码

```javascript
#!/usr/bin/env node
import { readFileSync, writeFileSync, existsSync, readdirSync } from 'node:fs';
import { join, basename } from 'node:path';

const KNOWN_FRAMEWORKS = { /* 见 3.1.4 */ };

/**
 * 从 rawDescription 和 readmeHead 合成 description。
 * - rawDescription 非空则直接使用
 * - 否则从 readmeHead 提取第一段非标题文本
 * - 截取前 200 字符（在词边界处截断），超出加 "..."
 * - 均为空则返回 "No description available"
 */
function synthesizeDescription(rawDescription, readmeHead) {
  // 1. rawDescription 非空则直接返回
  if (rawDescription) return rawDescription;

  // 2. 从 readmeHead 提取
  if (readmeHead) {
    const lines = readmeHead.split('\n');
    const paragraphLines = [];
    let inParagraph = false;
    for (const line of lines) {
      const trimmed = line.trim();
      // 跳过标题行（以 # 开头）
      if (/^#{1,6}\s/.test(trimmed)) continue;
      // 跳过空行
      if (trimmed === '') {
        if (inParagraph) break; // 段落结束
        continue;
      }
      // 第一个非空非标题行开始收集
      inParagraph = true;
      paragraphLines.push(trimmed);
    }
    const paragraph = paragraphLines.join(' ');
    if (paragraph) {
      if (paragraph.length <= 200) return paragraph;
      // 在词边界处截断
      const truncated = paragraph.slice(0, 200);
      const lastSpace = truncated.lastIndexOf(' ');
      return (lastSpace > 0 ? truncated.slice(0, lastSpace) : truncated) + '...';
    }
  }

  // 3. 均为空
  return 'No description available';
}

function main() {
  const [,, projectRoot, outputPath] = process.argv;
  if (!projectRoot || !outputPath) {
    process.stderr.write('Usage: extract-metadata.mjs <projectRoot> <outputPath>\n');
    process.exit(1);
  }

  // 1. package.json
  let name = null;
  let rawDescription = '';
  const depKeys = [];
  const pkgPath = join(projectRoot, 'package.json');
  if (existsSync(pkgPath)) {
    try {
      const pkg = JSON.parse(readFileSync(pkgPath, 'utf-8'));
      name = pkg.name || null;
      rawDescription = pkg.description || '';
      if (pkg.dependencies) depKeys.push(...Object.keys(pkg.dependencies));
      if (pkg.devDependencies) depKeys.push(...Object.keys(pkg.devDependencies));
    } catch {
      process.stderr.write('Warning: extract-metadata: package.json exists but is not valid JSON — skipping\n');
    }
  }

  // 2. name 回退到目录名
  if (!name) {
    name = basename(projectRoot);
  }

  // 3. README.md
  const readmeHead = extractReadmeHead(projectRoot);

  // 4. 框架匹配
  const frameworks = [];
  for (const [depKey, friendlyName] of Object.entries(KNOWN_FRAMEWORKS)) {
    if (depKeys.includes(depKey)) {
      frameworks.push(friendlyName);
    }
  }

  // 5. 基础设施检测
  if (detectDockerfile(projectRoot)) frameworks.push('Docker');
  if (detectGitHubActions(projectRoot)) frameworks.push('GitHub Actions');
  if (detectGitLabCI(projectRoot)) frameworks.push('GitLab CI');

  // 6. languages
  const scanResultPath = join(projectRoot, '.understand-anything-trae', 'intermediate', 'scan-result.json');
  const languages = extractLanguages(scanResultPath);

  // 7. 输出
  const description = synthesizeDescription(rawDescription, readmeHead);
  const result = { name, description, rawDescription, readmeHead, frameworks, languages };
  writeFileSync(outputPath, JSON.stringify(result, null, 2) + '\n');
  process.stderr.write(`extract-metadata: name=${name} frameworks=${frameworks.length} languages=${languages.length}\n`);
}

main();
```

### Step A (deterministic `extract-metadata.mjs`) -- Extract project metadata

Invoke the bundled metadata extraction script. It reads project manifests and README to produce name, description, frameworks, and languages — deterministically, without LLM inference.

```bash
node $PLUGIN_ROOT/skills/understand/extract-metadata.mjs \

Output JSON shape:

The script:

```

#### 3.3.2 Phase 2 变更

Phase 2 的 description 合成逻辑需要适配新的 Step A 输出格式：

**当前**：LLM 在 Step A 中直接合成 `rawDescription`，Phase 2 再加工为 `description`

**变更后**：`extract-metadata.mjs` 输出 `rawDescription`（直接从 `package.json` 读取）和 `description`（由 `synthesizeDescription` 从 `rawDescription` 或 `readmeHead` 首段合成）

### 3.4 精度影响分析

| 字段 | LLM 行为 | 确定性行为 | 精度影响 |
|------|---------|-----------|---------|
| `name` | 从 `package.json` 合成，可能格式化 | 直接读取 `package.json` `name` 字段 | **更准确**：无 LLM 格式化差异 |
| `description` | LLM 从 `package.json` + README 合成，可能改写 | `synthesizeDescription` 从 `rawDescription` 或 `readmeHead` 首段提取，200 字符截断 | **等价或更准确**：当 `package.json` 有 `description` 时完全一致；无 `description` 时从 README 首段提取（与 LLM 结果接近） |
| `rawDescription` | 从 `package.json` 合成，可能改写 | 直接读取 `package.json` `description` 字段 | **更准确**：无 LLM 改写 |
| `readmeHead` | 读取前 ~10 行 | 读取前 10 行 | **等价** |
| `frameworks` | LLM 从依赖列表推断 | 精确匹配依赖名 | **更可靠**：无 LLM 遗漏或误推断 |
| `languages` | LLM 从 manifests + Step B 统计合成 | 从 `scan-result.json` 的 `byLanguage` 键集合生成 | **等价**：来源相同 |

**精度说明**：当 `package.json` 无 `description` 且有 `README.md` 时，`synthesizeDescription` 从 README 首段提取（跳过标题行和空行，截取前 200 字符），与 LLM 合成结果接近。
